# 🌊 마이페이지 내 이미지 검증 기록 조회 API 흐름도

## 1. 요청 수신 (Router 단계)
클라이언트(프론트엔드)가 `GET /api/v1/auth/me/history/image` 엔드포인트로 요청을 보내. 
이때 헤더에는 인증을 위한 JWT 토큰이 들어있고, URL 뒤에는 몇 페이지를 볼 건지(`page`), 몇 개씩 볼 건지(`size`), 파일명 검색어(`keyword`), 결과 필터(`result_type`) 같은 옵션들이 파라미터로 같이 넘어와.

## 2. 사용자 인증 (Auth/Middleware 단계)
서버에 요청이 들어오면 가장 먼저 요금소(Middleware 또는 Dependency)를 거쳐. 
여기서 넘어온 JWT 토큰이 유효한지 검사하고, 진짜 우리 유저가 맞다면 토큰 안에서 이 요청을 보낸 사람의 고유 번호(`user_id`)를 쏙 뽑아내. 

## 3. 데이터 조인 및 1차 걸러내기 (Repository 단계)
이제 DB 창고로 가서 데이터를 꺼낼 차례야. 
* **테이블 합체:** 이미지 기본 정보가 있는 `images` 테이블과 최종 분석 결과가 있는 `image_final_detection_results` 테이블을 `image_id`를 기준으로 딱 붙여서(Join) 하나의 큰 표로 만들어.
* **내 것만 보기:** 합쳐진 데이터 중에서 아까 2번에서 뽑아낸 `user_id`와 일치하는 데이터(즉, 내가 업로드한 사진)만 일단 싹 걷어와.

## 4. 상세 조건 필터링 (Repository 단계)
사용자가 검색어나 필터를 걸었다면 여기서 쳐낼 건 쳐내야 해.
* **파일명 검색:** `keyword`가 넘어왔다면, 파일명(`filename`)에 그 단어가 포함된 데이터만 남겨.
* **결과 필터링:** `result_type`이 'ai'라면 최종 판별 결과가 AI인 것(`final_is_ai = true`)만, 'real'이라면 자연인 것(`final_is_ai = false`)만 남기고 나머지는 버려.

## 5. 정렬 및 자르기 (Repository 단계)
필터링까지 다 끝난 알짜배기 데이터들을 화면에 보여줄 수 있게 다듬어.
* **개수 세기:** 남은 전체 데이터가 총 몇 개인지 세어둬. (나중에 화면에 '총 N건'이라고 보여주거나 페이지 수를 계산할 때 써)
* **최신순 정렬:** 업로드 날짜(`created_at`)를 기준으로 가장 최근에 검증한 것부터 맨 위에 오도록 내림차순으로 줄을 세워.
* **페이징(자르기):** 줄 선 데이터 중에서 클라이언트가 요청한 `page`와 `size`에 맞춰서 딱 필요한 구간만 뚝 잘라서 가져와. (예: 1페이지에 10개씩이면 맨 위에서 10개만)

## 6. 예쁘게 포장하기 (Service 단계)
DB에서 가져온 투박한 날것의 데이터를 프론트엔드가 딱 받기 좋은 형태(API 명세서에 적힌 스펙)로 포장해.
* 전체 데이터 개수를 화면에 보여줄 개수로 나눠서 총 몇 페이지(`total_pages`)가 나오는지 계산해.
* 파일명, 판별 결과(`is_ai`), 확률(`ai_probability`), 날짜 등을 `histories`라는 이름의 리스트 안에 예쁘게 차곡차곡 담아.

## 7. 최종 배달 (Router 단계)
포장이 다 끝난 데이터를 JSON 형태(`success: true`, `data: { ... }`)로 만들어서 클라이언트에게 200 OK 응답과 함께 배달해 주면 이 API의 역할은 끝이야!
